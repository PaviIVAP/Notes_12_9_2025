Call a superclass method (especially when it's overridden)

Refer to superclass fields (if shadowed)

    }

    }

WHICH TO USE WHEN

‚úÖ List Implementations
ArrayList ‚Äì Maintains insertion order. Backed by an array, so it offers fast access by index, but slower insertions/removals in the middle.

LinkedList ‚Äì Maintains insertion order. Uses a doubly linked list, making it efficient for adding/removing elements at the beginning, middle, or end, but slower for random access compared to ArrayList.

‚úÖ Set Implementations
HashSet ‚Äì Does not maintain order, does not allow duplicates, and offers fast access due to hashing.

LinkedHashSet ‚Äì Maintains insertion order, does not allow duplicates, and is slightly slower than HashSet due to the overhead of maintaining order.

‚úÖ Map Implementations
HashMap ‚Äì Stores key-value pairs with no guaranteed order. Offers fast access via hashing.

LinkedHashMap ‚Äì Stores key-value pairs and preserves insertion order. Slightly slower than HashMap due to order tracking.

HASHING


Hashing is the process of converting a key into a hash code using a hash function. This hash code is then used to determine where the corresponding value should be stored or retrieved in a data structure like a HashMap or HashSet

Every time you want to retrieve a value from a HashMap, the key you provide is hashed again using the same hash function. This freshly computed hash code is then used to locate the bucket (or memory slot) where the value is stored.

Why ArrayList Is Different
It‚Äôs backed by a resizable array, which means:

Fast access by index (O(1)).

But slower insertions/removals in the middle (O(n)), because elements have to shift.

HOW A CLASS AND MEMORY IS STORED 


public class Customer {
    static String bankName = "ICICI"; // Static variable ‚Üí Method Area
    String name;                      // Instance variable ‚Üí Heap

    public Customer(String name) {
        this.name = name;            // Constructor logic ‚Üí Stack + Heap
    }

    public void greet() {
        System.out.println("Hello, " + name); // Method call ‚Üí Stack
    }

    public static void showBank() {
        System.out.println("Bank: " + bankName); // Static method ‚Üí Method Area
    }
}


Step 1: Class Loading
java
Customer.showBank();
JVM loads Customer.class into the Method Area.

bankName and showBank() are stored here.

No object is created yet ‚Äî just the blueprint.

üì¶ Memory Used:

Method Area ‚Üí Customer class metadata + static method + static variable

üß± Step 2: Object Creation
java
Customer c1 = new Customer("Pavitha");
new Customer("Pavitha") creates an object in the Heap.

name = "Pavitha" is stored inside that object.

c1 is a reference variable, stored in the Stack (inside main() method frame).

üì¶ Memory Used:

Heap ‚Üí Object with field name = "Pavitha"

Stack ‚Üí Reference c1 pointing to the Heap object

üß± Step 3: Method Invocation
java
c1.greet();
JVM pushes a new Stack frame for greet() method.

Inside that frame, it accesses name from the Heap via c1.

üì¶ Memory Used:

Stack ‚Üí Frame for greet() method

Heap ‚Üí Still holds the object

Method Area ‚Üí Already loaded

üßπ Step 4: Cleanup (Garbage Collection)
java
c1 = null; // Reference removed
Now, no variable points to the object in the Heap.

JVM marks it as eligible for garbage collection.

At some point, the Garbage Collector will reclaim that memory.

üì¶ Memory Freed:

Heap ‚Üí Object is removed

Stack ‚Üí Frame for main() eventually ends

Method Area ‚Üí Class remains loaded until JVM shuts down
